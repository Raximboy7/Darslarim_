Modul 22 — DevOps (Docker va CI/CD) | Dars 163

Bu Darsda Nima Qilamiz?

Bu modulning yakuniy proyekti. Biz shu paytgacha o'rgangan hamma narsani — Docker, docker-compose, GitHub Actions, secrets va monitoringni — bitta to'liq deploy pipelinega birlashtiramiz. Maqsad: kodni yozganingizdan keyin u avtomatik tarzda tekshirilib, qadoqlab, serverga joylanishi (deploy) va kuzatilishi.
Pipeline (oqim) nima?

Pipeline — bu kodingiz boshidan oxirigacha o'tadigan avtomatik bosqichlar zanjiri. Siz faqat kod yozasiz va git push qilasiz — qolgan hammasi o'zi bajariladi.

To'liq Oqim: 5 Bosqich

Bizning pipeline 5 bosqichdan iborat. Har bosqich faqat oldingisi muvaffaqiyatli tugagandagina ishga tushadi.
#BosqichNima bo'ladi
1CommitKodni yozib, GitHub'ga git push qilasiz
2TestGitHub Actions testlarni avtomatik ishga tushiradi
3BuildDocker image tayyorlanadi (qadoqlanadi)
4DeployImage serverga joylanadi va ishga tushadi
5MonitorHealthcheck va loglar holatni kuzatadi
kod → commit → test → build → deploy → monitor

1-Bosqich: Dockerfile Yozish

Dockerfile — bu ilovangizni qadoqlash uchun retsept. U Python o'rnatadi, kutubxonalarni yuklaydi va ilovani ishga tushiradigan buyruqni belgilaydi.
# Tayyor Python image'dan boshlaymiz
FROM python:3.11-slim

# Ish papkasini belgilaymiz
WORKDIR /app

# Avval kutubxonalar ro'yxatini ko'chiramiz
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Endi qolgan kodni ko'chiramiz
COPY . .

# Ilovani ishga tushiramiz
CMD ["python", "app.py"]
# requirements.txt ni alohida ko'chirish — keshdan unumli foydalanish uchun

2-Bosqich: docker-compose bilan Xizmatlar

Ilova ko'pincha bir nechta xizmatdan iborat bo'ladi (masalan ilova + ma'lumotlar bazasi). docker-compose.yml ularni bitta joyda boshqaradi.
services:
  web:
    build: .
    ports:
      - "8000:8000"
    environment:
      - DB_URL=${DB_URL}
  db:
    image: postgres:16
    environment:
      - POSTGRES_PASSWORD=${DB_PASS}
# ${DB_URL} va ${DB_PASS} — maxfiy qiymatlar kodda emas, .env yoki secrets'dan keladi

3-Bosqich: GitHub Actions Workflow

Bu eng muhim qism. .github/workflows/deploy.yml faylida test → build → deploy bosqichlarini yozamiz. Har git pushda bu avtomatik ishlaydi.
name: Deploy Pipeline
on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest
# 1-job: testlar. main branch'ga push bo'lganda ishga tushadi
  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t myapp:latest .

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: ./deploy.sh
# needs: test → build faqat test o'tgach ishlaydi. needs: build → deploy faqat build o'tgach

4-Bosqich: Secrets (Maxfiy Ma'lumotlar)

Parol, API kalit yoki server ma'lumotlarini hech qachon kodga yozmang. Ularni GitHub'da Settings → Secrets bo'limiga saqlaysiz va workflow ichida ${{ secrets.NOM }} orqali olasiz.
  deploy:
    steps:
      - run: ssh deploy@${{ secrets.SERVER_IP }}
        env:
          DB_PASS: ${{ secrets.DB_PASS }}
# secrets.SERVER_IP va secrets.DB_PASS loglar ichida ham yashirin (***) ko'rinadi
Xavfsizlik qoidasi

Parolni kodga yozsangiz va GitHub'ga push qilsangiz — u butun dunyoga ochiq bo'ladi. Har doim secrets ishlating.

5-Bosqich: Monitoring

Deploy tugagach ish bitmaydi. Ilova ishlayotganini healthcheck va loglar orqali kuzatamiz. Agar xato chiqsa — darhol bilamiz.
# Healthcheck: ilova "tirik"mi?
HEALTHCHECK --interval=30s \
  CMD curl -f http://localhost:8000/health || exit 1
# Har 30 soniyada /health tekshiriladi. Javob bermasa konteyner "unhealthy" deb belgilanadi
>>> docker ps
STATUS: Up 5 minutes (healthy)
Eslab Qoling

• Pipeline = avtomatik bosqichlar zanjiri
• needs: bosqichlar tartibini belgilaydi
• Maxfiy ma'lumotlar — faqat secretsda
• Bitta bosqich yiqilsa — keyingilari ishlamaydi
• Deploy'dan keyin monitoring shart
