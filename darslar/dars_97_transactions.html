Transactions va Yaxlitlik

Modul 14 — Database va SQL | Dars 97

Transaction Nima?

Transaction (tranzaksiya) — bu bir nechta SQL amalini bitta butun sifatida bajarishdir. Ular yo hammasi birga bajariladi, yo hammasi bekor qilinadi. Yarim qolish bo'lmaydi.
Tasavvur qiling: Aliyev hisobidan Valiyev hisobiga pul o'tkazyapsiz. Bu ikki amal: Aliyevdan ayirish va Valiyevga qo'shish. Agar birinchisi bajarilib, ikkinchisi xato bersa — pul "yo'qoladi". Transaction aynan shuni oldini oladi: ikkalasi birga bo'ladi yoki ikkalasi ham bekor qilinadi.

BEGIN, COMMIT va ROLLBACK

Transaction uch buyruq bilan boshqariladi:
BuyruqVazifasi
BEGINTransactionni boshlaydi (shu yerdan keyingi amallar guruhlanadi)
COMMITBarcha o'zgarishlarni tasdiqlaydi va saqlaydi
ROLLBACKBarcha o'zgarishlarni bekor qiladi, boshlang'ich holatga qaytaradi
BEGIN;
UPDATE hisoblar SET balans = balans - 100 WHERE id = 1;
UPDATE hisoblar SET balans = balans + 100 WHERE id = 2;
COMMIT;
# Ikki amal birga: 1-hisobdan ayirildi, 2-hisobga qo'shildi.
# COMMIT — hammasi muvaffaqiyatli, saqlaymiz.
BEGIN;
UPDATE hisoblar SET balans = balans - 100 WHERE id = 1;
-- xatolik yuz berdi!
ROLLBACK;
# ROLLBACK — xato bo'ldi, hamma narsani bekor qilamiz.
# 1-hisob balansi o'zgarmaydi, hech narsa yo'qolmaydi.

ACID Xususiyatlari

Yaxshi transaction ACID qoidalariga amal qiladi. Bu to'rt so'zning bosh harflari:
HarfNomiSodda tushuntirish
AAtomicity (Bo'linmaslik)Hammasi birga bajariladi yoki hech narsa bajarilmaydi
CConsistency (Izchillik)Ma'lumotlar har doim to'g'ri qoidalarga mos qoladi
IIsolation (Ajratilganlik)Bir vaqtda ishlayotgan transactionlar bir-biriga xalal bermaydi
DDurability (Mustahkamlik)COMMIT qilingach, ma'lumot saqlanadi (hatto tok o'chsa ham)
Eslatma

ACID — bu ma'lumotlaringiz ishonchli va yaxlit bo'lishi kafolati. Bank, do'kon, ombor kabi tizimlarda bu juda muhim.

Foreign Key — Jadvallar Bog'liqligi

Foreign key (tashqi kalit) — bir jadvalni boshqasiga bog'laydi va bog'liqlik yaxlitligini saqlaydi. Masalan, har bir buyurtma mavjud bo'lgan mijozga tegishli bo'lishi shart.
CREATE TABLE buyurtmalar (
    id INTEGER PRIMARY KEY,
    mijoz_id INTEGER,
    FOREIGN KEY (mijoz_id) REFERENCES mijozlar(id)
);
# mijoz_id mijozlar jadvalidagi mavjud id ga ishora qilishi shart.
# Mavjud bo'lmagan mijozga buyurtma yozib bo'lmaydi.

Constraints — Cheklovlar

Constraint (cheklov) — ustunlarga qo'yiladigan qoidalar. Ular noto'g'ri ma'lumot kiritilishini oldini oladi.
CheklovVazifasi
NOT NULLUstun bo'sh (NULL) bo'lishi mumkin emas — qiymat shart
UNIQUEQiymat takrorlanmasligi shart (masalan, email)
PRIMARY KEYAsosiy kalit — NOT NULL va UNIQUE birgalikda
CHECKShartni tekshiradi (masalan, balans >= 0)
CREATE TABLE foydalanuvchilar (
    id INTEGER PRIMARY KEY,
    ism TEXT NOT NULL,
    email TEXT UNIQUE
);
# ism bo'sh bo'lishi mumkin emas (NOT NULL).
# email takrorlanmasligi shart (UNIQUE).
Eslab Qoling

• BEGIN boshlaydi, COMMIT tasdiqlaydi, ROLLBACK bekor qiladi
• ACID — Atomicity, Consistency, Isolation, Durability
• FOREIGN KEY jadvallar bog'liqligini saqlaydi
• NOT NULL va UNIQUE noto'g'ri ma'lumotni rad etadi

Misol 1: Bank O'tkazmasi (COMMIT vs ROLLBACK)

Aliyevdan Valiyevga pul o'tkazamiz. COMMIT bosilsa o'zgarish saqlanadi, ROLLBACK bosilsa boshlang'ich holatga qaytadi.
Aliyev (id=1)
500
Valiyev (id=2)
300

Misol 2: Xato Bo'lganda ROLLBACK

Agar o'tkaziladigan summa balansdan ko'p bo'lsa, ikkinchi amal xato beradi va tizim avtomatik ROLLBACK qiladi — hech narsa o'zgarmaydi.

Misol 3: UNIQUE Cheklovi Buzilganda

email ustuni UNIQUE. Bazada allaqachon mavjud emailni qayta kiritsangiz, baza uni rad etadi.
